# EF Core Migration Conventions

## Critical Rules

- **Never hand-edit auto-generated migration files** — the `Up()` and `Down()` methods are generated by EF Core; manual edits almost always break rollback integrity.
- **Always review** the generated migration diff in source control before applying it to any shared database.
- **Name migrations descriptively** — the name becomes permanent history.

```
// ✅ Correct migration names
AddEmployeeSoftDelete
AddLeaveStatusIndex
AddDepartmentEntityWithForeignKey

// ❌ Incorrect migration names
Migration1
FixBug
Update20240411
```

## Workflow Commands

This project uses the separate Infrastructure and API projects. Always specify `--project` and `--startup-project`.

### Prerequisites
```bash
# Verify EF Core tools are installed
dotnet ef --version

# Install if missing
dotnet tool install --global dotnet-ef
```

### Add a Migration
```bash
dotnet ef migrations add {DescriptiveName} \
  --project src/LeaveManagement.Infrastructure \
  --startup-project src/LeaveManagement.API
```

### Review Before Applying
After generating, open the new file under `src/LeaveManagement.Infrastructure/Migrations/` and check:
- Expected table/column changes are present
- No unexpected drops (a column rename often appears as drop + add)
- `Down()` method is the correct inverse of `Up()`

### Apply to Database
```bash
dotnet ef database update \
  --project src/LeaveManagement.Infrastructure \
  --startup-project src/LeaveManagement.API
```

### Rollback to a Previous Migration
```bash
# Rolls back to (but does not undo) the named migration
dotnet ef database update {PreviousMigrationName} \
  --project src/LeaveManagement.Infrastructure \
  --startup-project src/LeaveManagement.API
```

### Remove Last Migration (unapplied only)
```bash
dotnet ef migrations remove \
  --project src/LeaveManagement.Infrastructure \
  --startup-project src/LeaveManagement.API
```

> ⚠️ `migrations remove` only works if the migration has **not** been applied to any database. If it has, rollback first.

## Current State: EnsureCreatedAsync

This project currently uses `Database.EnsureCreatedAsync()` (in `StartupExtensions.cs`) for dev-only schema creation. This is incompatible with migrations — to switch:

1. Remove the `EnsureCreatedAsync()` call
2. Run `dotnet ef migrations add InitialCreate ...` to capture the current schema
3. Apply with `dotnet ef database update ...`
4. Migrations will now manage all schema evolution

## What NOT to Do

- ❌ Edit generated migration files (`Up()` / `Down()` / `BuildTargetModel()`)
- ❌ Apply migrations directly to production without reviewing the diff
- ❌ Delete migration files without first rolling back the database
- ❌ Mix `EnsureCreatedAsync()` and migrations — they will conflict
